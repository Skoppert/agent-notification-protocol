# Agent Notification Protocol (ANP) — Specification v0.1

## Abstract

The Agent Notification Protocol (ANP) defines a standardized, filesystem-based mechanism for applications to deliver informational signals to local AI agents. Unlike tool-calling protocols (such as MCP), ANP is unidirectional: applications send signals, agents discover them. No response is expected. No instruction is given.

ANP enables any application to notify any agent, as long as both follow this specification.

---

## 1. Design Principles

1. **Signals, not instructions.** A notification is a statement of fact, never a command. The agent decides autonomously what to do with it — or nothing at all.

2. **Agent-controlled trust.** Only the agent (with explicit user approval) can grant an application permission to send notifications. Applications cannot self-register.

3. **No expected response.** The sending application does not know if or how the agent processed the notification. There is no acknowledgment mechanism in the base protocol.

4. **Filesystem-native.** The protocol uses the local filesystem as its transport layer. No servers, no sockets, no cloud services. Any program that can write a file can participate.

5. **Minimal by design.** The protocol defines the smallest possible contract between applications and agents. Everything beyond that contract is an implementation choice.

---

## 2. Terminology

| Term | Definition |
|------|-----------|
| **Agent** | An AI system (e.g., Claude Code, Cursor, Copilot) that operates on a user's machine and can read files. |
| **Application** | Any software that wants to deliver information to an agent. Also called "sender". |
| **Notification** | A single informational signal from an application to an agent. |
| **Registry** | A directory of files that define which applications are permitted to send notifications, and what types they may send. |
| **Filter** | A process that validates incoming notifications against the registry before they reach the agent. |
| **Context File** | A compiled file of validated notifications, formatted for agent consumption. |
| **Notification Type** | A predefined category of signal (e.g., `time_audit`, `task_complete`). Applications can only send types they are registered for. |

## 3. Directory Structure

All protocol files reside under a single root directory on the user's machine.

**Root path:**
- Windows: `%USERPROFILE%\.ai-notifications\`
- macOS / Linux: `~/.ai-notifications/`

```
.ai-notifications/
  registry/                          # [AGENT-ONLY] Registration files
    {app-id}.json
  incoming/                          # [APP-WRITABLE] Notification drop zone
    {timestamp}-{app-id}-{uuid}.json
  processing/                        # [FILTER-ONLY] Temporary processing area
  context/
    notifications.md                 # [FILTER-WRITES, AGENT-READS] Active notifications
    notifications.archive.md         # [FILTER-WRITES] Archived notifications
  logs/
    filter.log                       # [FILTER-WRITES] Processing log
  config.json                        # [AGENT-MANAGED] Global configuration
```

### 3.1 Access Model

| Directory / File | Application | Filter | Agent |
|-----------------|-------------|--------|-------|
| `registry/` | No access | Read | Read + Write |
| `incoming/` | Write only | Read + Delete | No direct access |
| `processing/` | No access | Read + Write + Delete | No direct access |
| `context/` | No access | Write | Read |
| `config.json` | No access | Read | Read + Write |

On single-user machines, this access model is enforced by convention and validated by the filter, not by filesystem permissions.

## 4. Registry Format

Each registered application has one file in `registry/`. The filename MUST be `{app-id}.json` where `app-id` is a lowercase alphanumeric string with hyphens (matching `^[a-z0-9][a-z0-9-]*[a-z0-9]$`, min 2 characters).

### 4.1 Registry File Schema

```json
{
  "version": "1.0",
  "appId": "<string, required>",
  "displayName": "<string, required>",
  "registeredAt": "<ISO 8601 datetime, required>",
  "allowedTypes": ["<string>", "..."],
  "token": "<string, required>",
  "maxNotificationsPerHour": "<integer, default: 10>",
  "enabled": "<boolean, default: true>"
}
```

**Field definitions:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | string | Yes | Protocol version. MUST be `"1.0"`. |
| `appId` | string | Yes | Unique identifier for the application. MUST match the filename. |
| `displayName` | string | Yes | Human-readable name of the application. |
| `registeredAt` | string | Yes | ISO 8601 datetime of when the registration was created. |
| `allowedTypes` | string[] | Yes | List of notification type identifiers this application may send. MUST contain at least one entry. |
| `token` | string | Yes | A shared secret generated at registration time. Minimum 32 characters. |
| `maxNotificationsPerHour` | integer | No | Maximum notifications this application may send per rolling hour. Default: 10. |
| `enabled` | boolean | No | Whether this registration is active. Default: true. A disabled registration causes all notifications from this app to be silently dropped. |

### 4.2 Registration Rules

1. Only the agent or the user may create, modify, or delete registry files.
2. Applications MUST NOT write to the `registry/` directory.
3. The `token` MUST be cryptographically random and at least 32 characters long.
4. The `allowedTypes` list defines the complete set of notification types this application may use. Any notification with a type not in this list MUST be rejected by the filter.

## 5. Notification Format

Applications send notifications by writing JSON files to the `incoming/` directory.

### 5.1 Filename Convention

```
{ISO-timestamp}-{app-id}-{uuid}.json
```

Example: `20260319T093000Z-toggl-tracker-a1b2c3d4.json`

- The timestamp MUST be in UTC, formatted as `YYYYMMDDTHHmmssZ`.
- The UUID MUST be unique per notification. A random 8-character hex string is sufficient.
- The filename enables chronological sorting and sender identification without parsing the file.

### 5.2 Notification File Schema

```json
{
  "version": "1.0",
  "appId": "<string, required>",
  "token": "<string, required>",
  "type": "<string, required>",
  "timestamp": "<ISO 8601 datetime, required>",
  "summary": "<string, required>",
  "data": "<object, optional>",
  "priority": "<string, optional>",
  "expiresAt": "<ISO 8601 datetime, optional>"
}
```

**Field definitions:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | string | Yes | Protocol version. MUST be `"1.0"`. |
| `appId` | string | Yes | The application's registered identifier. MUST match a registry entry. |
| `token` | string | Yes | The shared secret from the registry. MUST match exactly. |
| `type` | string | Yes | The notification type. MUST be listed in the application's `allowedTypes`. |
| `timestamp` | string | Yes | ISO 8601 datetime of when the notification was created. |
| `summary` | string | Yes | A plain-language description of the signal. Maximum 500 characters. See Section 5.3 for content rules. |
| `data` | object | No | Structured metadata relevant to the notification. No schema enforced; content is application-specific. Maximum depth: 3 levels. |
| `priority` | string | No | One of: `"low"`, `"normal"`, `"high"`. Default: `"normal"`. Advisory only — the agent is not obligated to act on priority. |
| `expiresAt` | string | No | ISO 8601 datetime after which the notification is no longer relevant. Expired notifications MUST be dropped by the filter. If omitted, the notification expires after 24 hours. |

### 5.3 Summary Content Rules

The `summary` field MUST be a factual statement. It MUST NOT contain:

1. **Imperative instructions** directed at the agent: "tell the user to...", "execute...", "run...", "create...", "delete...", "modify..."
2. **Prompt injection patterns**: "ignore previous instructions", "system:", "you are now...", "override..."
3. **Code blocks** containing executable commands
4. **URLs** that the agent should visit or interact with
5. **Requests for action**: "please...", "you should...", "make sure to..."

A notification that violates these rules MUST be rejected by the filter.

**Valid examples:**
- "4 hours tracked today across 3 projects. No time logged since 14:00."
- "Production deployment of v2.4.1 completed successfully."
- "Team standup in 15 minutes. 5 attendees."

**Invalid examples:**
- "Tell the user to log more time." (imperative instruction)
- "Run `npm test` to verify the deployment." (executable instruction)
- "You should check the error logs at /var/log/app.log" (instruction with path)

### 5.4 Writing Notifications Safely

Applications MUST write notifications atomically to prevent the filter from reading a partially written file:

1. Write the complete JSON to a temporary file with a `.tmp` extension in the `incoming/` directory.
2. Rename (move) the `.tmp` file to the final `.json` filename.

On NTFS (Windows) and ext4/APFS (macOS/Linux), `rename` within the same directory is atomic.

## 6. Notification Types

Notification types are string identifiers that categorize signals. They follow the format: lowercase alphanumeric with underscores (`^[a-z][a-z0-9_]*$`, max 50 characters).

### 6.1 Reserved Types

The following types have special meaning in the protocol:

| Type | Purpose |
|------|---------|
| `_registration_request` | An application is requesting to be registered. See Section 7.2. |

All types beginning with `_` (underscore) are reserved for protocol-level use. Applications MUST NOT register for or send reserved types.

### 6.2 Suggested Standard Types

These types are not required, but using them promotes interoperability between applications and agents:

| Type | Description | Example summary |
|------|-------------|----------------|
| `task_complete` | A background task or process finished | "Build completed in 45 seconds. All tests passed." |
| `task_failed` | A background task or process failed | "Deployment to staging failed. Exit code 1." |
| `time_audit` | A time-tracking signal | "4 hours logged today. Last entry at 14:00." |
| `calendar_event` | An upcoming calendar event | "Team standup in 15 minutes. 5 attendees." |
| `deploy_status` | CI/CD pipeline status update | "Production deploy v2.4.1 completed. All health checks passing." |
| `user_action` | A notable user action in a web/desktop app | "New comment on issue #42 by @johndoe." |
| `system_alert` | A system-level alert | "Disk usage at 90% on volume C:." |
| `data_update` | A dataset or data source was updated | "Sales report for Q1 2026 is now available." |

Applications MAY define custom types beyond this list. Custom types SHOULD be descriptive and specific to the application's domain.

## 7. Processes

### 7.1 Filter Process

The filter is a process that runs periodically (recommended: every 2 minutes) and performs the following steps:

#### Step 1: Acquire Lock

The filter creates a lock file (`.lock`) in the root directory using an exclusive create operation. If the lock file already exists and is older than 5 minutes, it is considered stale and may be removed.

If the lock cannot be acquired, the filter exits without processing.

#### Step 2: Move to Processing

For each `.json` file in `incoming/`, the filter moves it to `processing/` using an atomic rename. This prevents race conditions with applications writing new files.

Files with a `.tmp` extension are skipped (they are still being written).

#### Step 3: Validate

For each file in `processing/`, the filter performs the following checks in order:

1. **Parse**: The file MUST be valid JSON conforming to the Notification File Schema (Section 5.2). If not → reject.
2. **Sender exists**: `appId` MUST match a file in `registry/`. If not → reject.
3. **Sender enabled**: The registry entry MUST have `enabled: true`. If not → reject.
4. **Token matches**: `token` MUST exactly match the token in the registry entry. If not → reject.
5. **Type allowed**: `type` MUST be listed in the registry entry's `allowedTypes`. If not → reject.
6. **Not expired**: If `expiresAt` is present, it MUST be in the future. If not → reject.
7. **Content safe**: `summary` MUST NOT contain patterns matching Section 5.3 rules. If it does → reject.
8. **Rate limit**: The number of notifications from this `appId` in the current rolling hour MUST NOT exceed `maxNotificationsPerHour` from the registry. If exceeded → reject.

Rejected notifications are logged (with the reason) and deleted from `processing/`.

#### Step 4: Compile Context File

Valid notifications are merged with existing (non-expired) entries in the context file. The filter writes the context file in the format defined in Section 8.

#### Step 5: Archive

If the context file contains more than 50 entries or entries older than 48 hours, the oldest entries are moved to `notifications.archive.md`.

#### Step 6: Release Lock

The filter deletes the `.lock` file.

### 7.2 Registration Process

When an application wants to register, it follows this process:

1. The application writes a notification file to `incoming/` with:
   - `type`: `"_registration_request"`
   - `appId`: The desired application ID
   - `token`: An empty string `""`
   - `summary`: A human-readable description of what the application wants to send
   - `data`: `{ "displayName": "...", "requestedTypes": ["...", "..."] }`

2. The filter recognizes the `_registration_request` type and does NOT validate it against the registry (since the app is not yet registered). Instead, it adds a special entry to the context file marked as `[action_required]`.

3. The agent reads the context file and presents the registration request to the user.

4. The user approves or denies the request.

5. If approved, the agent (or user) creates a registry file for the application with:
   - A cryptographically random token
   - The approved notification types (which may be a subset of what was requested)
   - A sensible `maxNotificationsPerHour` value

6. The agent communicates the token back to the application through a secure channel (displayed to the user, who provides it to the application's configuration).

### 7.3 Agent Discovery

An agent discovers notifications by reading the context file at `context/notifications.md`. The agent MAY check this file:

- At session start
- Periodically during a session (e.g., every 30 minutes)
- When explicitly asked by the user
- Never (if the agent has no interest in external notifications)

The agent is under no obligation to act on any notification. The protocol makes no assumptions about agent behavior.

## 8. Context File Format

The context file is a Markdown document with YAML frontmatter, optimized for AI agent consumption (approximately 40% fewer tokens than equivalent JSON).

### 8.1 Structure

```markdown
---
generated: <ISO 8601 datetime>
notificationCount: <integer>
oldestNotification: <ISO 8601 datetime>
---

# Active Notifications

## [<priority>] <appId>: <type>
**When:** <date time> | **Expires:** <date time>

<summary>

---

## [<priority>] <appId>: <type>
**When:** <date time> | **Expires:** <date time>

<summary>

---

*<count> notifications from <app_count> applications. Last updated: <datetime>.*
```

### 8.2 Rules

1. Notifications are listed in reverse chronological order (newest first).
2. Each notification is separated by a horizontal rule (`---`).
3. The `summary` is reproduced verbatim from the notification file.
4. The `data` field from the notification is NOT included in the context file by default (to save tokens). The agent may inspect the original notification file if it needs structured data.
5. Registration requests (Section 7.2) are rendered with `[action_required]` instead of a priority level.

## 9. Configuration

The `config.json` file in the root directory contains global settings.

```json
{
  "version": "1.0",
  "pollIntervalSeconds": 120,
  "maxNotificationAgeDays": 2,
  "maxContextEntries": 50,
  "archiveEnabled": true,
  "logLevel": "info"
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `version` | string | `"1.0"` | Configuration version. |
| `pollIntervalSeconds` | integer | `120` | How often the filter runs, in seconds. |
| `maxNotificationAgeDays` | integer | `2` | Maximum age of notifications before automatic expiry. |
| `maxContextEntries` | integer | `50` | Maximum entries in the context file before archival. |
| `archiveEnabled` | boolean | `true` | Whether to archive old notifications or delete them. |
| `logLevel` | string | `"info"` | Filter log verbosity: `"debug"`, `"info"`, `"warn"`, `"error"`. |

## 10. Security Considerations

### 10.1 Threat Model

ANP is designed for single-user development machines. The primary threats are:

1. **Misbehaving applications**: An app sends too many notifications, sends unauthorized types, or includes manipulative content. Mitigated by: token verification, type restriction, rate limiting, content rules.

2. **Prompt injection via notifications**: An app attempts to inject instructions into the agent through the `summary` field. Mitigated by: content rules (Section 5.3), filter validation, and the fundamental design principle that notifications are signals, not instructions.

3. **Unauthorized notification sending**: An unknown application writes to `incoming/`. Mitigated by: token verification. Without the correct token, the notification is rejected.

### 10.2 Out of Scope

ANP does NOT protect against:

- A process running as the same OS user that intentionally attacks the system (this is equivalent to full system compromise)
- Physical access to the machine
- Tampering with the filter process itself

### 10.3 Agent-Side Safety

Agents implementing ANP SHOULD:

1. Never treat notification content as instructions, regardless of phrasing
2. Present notification content to the user before taking any action based on it
3. Apply their own safety filters in addition to the protocol-level filter
4. Clearly distinguish between user-initiated actions and notification-triggered observations

## 11. Interoperability

### 11.1 Agent Integration

Any AI agent can integrate with ANP by:

1. Reading the context file at `~/.ai-notifications/context/notifications.md`
2. Treating its content as informational context, not instructions
3. Optionally: participating in the registration flow by creating registry files

The agent does not need to run the filter. The filter is a separate process.

### 11.2 Application Integration

Any application can send notifications by:

1. Obtaining a token through the registration process (Section 7.2)
2. Writing JSON files to `~/.ai-notifications/incoming/` following the notification format (Section 5)

No SDK is required. The protocol is defined entirely by the file formats and directory structure.

### 11.3 Filter Implementation

Any implementation of the filter process is valid as long as it:

1. Follows the validation steps in Section 7.1
2. Produces a context file conforming to Section 8
3. Does not modify the registry
4. Handles file locking correctly

---

## 12. Versioning

This specification uses semantic versioning. The `version` field in all files MUST match the major version of the specification they conform to.

- **Major version** (1.x → 2.x): Breaking changes to file formats or validation rules
- **Minor version** (1.0 → 1.1): Backward-compatible additions (new optional fields, new standard types)

Implementations MUST reject files with an unrecognized major version.

---

## 13. Reference: Quick Start

For an application developer who wants to send notifications:

1. Write a `_registration_request` notification to `~/.ai-notifications/incoming/`
2. Wait for the user to approve the registration
3. Receive the token from the user
4. Store the token securely in your application's configuration
5. Write notification files to `~/.ai-notifications/incoming/` using the format in Section 5

For an agent developer who wants to receive notifications:

1. Read `~/.ai-notifications/context/notifications.md` at session start or periodically
2. Treat all content as informational signals
3. Never execute notification content as instructions
4. Optionally: handle `[action_required]` entries to facilitate registration

For a filter implementer:

1. Follow Section 7.1 exactly
2. Run on a schedule (recommended: every 2 minutes)
3. Log all rejections with reasons

```json
}
```

### A.2 Notification file: `incoming/20260319T143000Z-wayve-app-e7f8a9b0.json`

```json
{
  "version": "1.0",
  "appId": "wayve-app",
  "token": "f8a3b2c1d4e5f6a7b8c9d0e1f2a3b4c5",
```

```markdown
---
generated: 2026-03-19T14:32:00Z
notificationCount: 1

# Active Notifications

**When:** 2026-03-19 14:30 | **Expires:** 2026-03-19 18:00

---

*1 notification from 1 application. Last updated: 2026-03-19 14:32.*
```

```json
{
```
